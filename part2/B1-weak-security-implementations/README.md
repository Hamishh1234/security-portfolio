# Activity B1 – Weak Security Implementations

## Objective
To discover 5 weak or vulnerable security implementations

## What I did
I looked at different examples of weak security that can happen in websites, accounts, networks and software. I tried to choose examples that are common and easy to understand.

## Findings
One weak security implementation is a website using HTTP instead of HTTPS. This is weak because the data being sent between the user and the website is not encrypted. If someone was able to intercept the traffic, they may be able to view information being sent.

Another weak implementation is a login page that does not appear to have CAPTCHA or any visible account lockout. If there is no limit on login attempts, someone could keep trying different passwords. This could make brute force attacks easier.

A WiFi network with no password or a weak password is also poor security. If anyone nearby can connect to the network, they may be able to use the internet connection or try to access devices connected to it.

Password reuse is another weak security practice. If someone uses the same password across multiple websites and one of those websites gets breached, the attacker could try the same password on other accounts.

Outdated software is also a security weakness. If software is not updated, it may still contain known vulnerabilities that attackers already know how to exploit.

## Evidence collected
I included screenshots showing examples of these weak security implementations in the evidence folder.

## Reflection
This activity showed me that weak security does not always come from complicated hacking. A lot of risks come from simple things like not using HTTPS, reusing passwords, having weak WiFi security, or not updating software. It made me realise that basic security mistakes can still cause serious problems.
